## Development Process

### Step 1: Find Next Work

```bash
bd ready --json -n 1
```

If no issues are ready, run `bd blocked` to diagnose and report what is stuck. Then STOP.

If an issue is ready, claim it:

```bash
bd update <id> --claim
bd show <id>
```

Read the full issue description. Understand WHY this issue exists and WHAT it unblocks.

### Step 2: Understand Context

Before writing any code:
- Read any files the issue description references
- Read the design-document.rst for architectural context
- Check what exists: `ls src/ backend/ frontend/ 2>/dev/null; cat pyproject.toml`
- If this issue depends on completed work, read that code first to understand the interfaces you must integrate with

### Step 3: Test-Driven Development

For each piece of functionality in the issue:

**3a. Write the test FIRST.**

Place tests in `tests/` mirroring `src/` structure:
- `src/evercurrent/ingestion/loader.py` -> `tests/test_ingestion/test_loader.py`

Every test file must have:
- Typed fixtures with explicit return types
- Descriptive test names: `test_<unit>_<scenario>_<expected_behavior>`
- Edge cases and error paths, not just happy paths
- No mocking unless crossing a system boundary (LLM API, filesystem, network)

**3b. Run the test - confirm it FAILS.**

```bash
uv run pytest tests/<relevant_test_file> -x -v 2>&1 | tail -20
```

**3c. Write the minimal implementation to make it pass.**

All production code goes in `src/evercurrent/`. Every module must have:

- **Google-style docstrings** on all public functions, classes, and modules:
  ```python
  def score_atom(atom: Atom, persona: Persona) -> ScoredAtom:
      """Compute composite relevance score for an atom-persona pair.

      Calculates weighted sum across five scoring dimensions:
      workstream proximity, role-type alignment, phase alignment,
      urgency, and social signal.

      Args:
          atom: The extracted information atom to score.
          persona: The persona whose relevance perspective to apply.

      Returns:
          ScoredAtom with composite score and per-dimension breakdown.

      Raises:
          ValueError: If persona scoring weights do not sum to 1.0.
      """
  ```

- **Full type annotations** on every function signature, variable where not obvious, and return type. Use `typing` module constructs. No `Any` unless interfacing with untyped external code.

- **Cyclomatic complexity <= 8 per function.** If a function needs more branches, decompose it. Prefer early returns, dict dispatch, and strategy patterns over nested if/elif chains.

- **High maintainability.** Functions should do one thing. Modules should have a clear single responsibility. Keep functions under 30 lines where possible.

**3d. Run the test - confirm it PASSES.**

```bash
uv run pytest tests/<relevant_test_file> -x -v
```

**3e. Repeat 3a-3d for each piece of the issue.**

### Step 4: Quality Gates

After all tests pass for the issue, run the full quality gate suite:

```bash
bash scripts/quality-gates.sh
```

If ANY gate fails:
1. Fix the issue
2. Re-run: `bash scripts/quality-gates.sh`
3. Do not proceed until all pass

If ruff reports errors, fix with `uv run ruff check --fix src/ tests/` then `uv run ruff format src/ tests/`, then re-run `bash scripts/quality-gates.sh`.

### Step 5: Close and Commit

```bash
# Close the issue
bd close <id> --reason="<1-line summary of what was built and tested>"

# Stage and commit only the files you changed
git add <specific files>
git commit -m "<descriptive message>"
```

### Step 6: Check What Unblocked

```bash
bd ready --json -n 3
```

Report what is now available. Then loop back to Step 1 for the next issue.

## Hard Rules

- NEVER skip writing tests first. If you catch yourself writing implementation before a failing test exists, stop and write the test.
- NEVER leave a quality gate failing. Zero ruff errors. Zero type errors. Coverage >= 90%.
- NEVER write a function with cyclomatic complexity > 8. Decompose.
- NEVER use `Any` type without a comment justifying why.
- NEVER add dependencies not in the issue description without noting it in the issue via `bd note <id> "Added dependency X because Y"`.
- ALWAYS read the issue description fully before starting. The descriptions contain architectural rationale from the design document.
- ALWAYS claim the issue before starting work (`bd update <id> --claim`).
- ALWAYS close the issue with a reason when done.
- If you encounter ambiguity in the issue description, check the design-document.rst. If still ambiguous, add a note to the issue and make a reasonable decision.
